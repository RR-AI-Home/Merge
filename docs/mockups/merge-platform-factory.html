Design rule

No runtime theme switching

What stays shared

Core game math, persistence model, engine tests, board UX components, economy hooks, analytics schema, purchase plumbing, and liveops tools.

What changes per game

App name, assets, item chains, producer fiction, progression map, onboarding copy, store listing, tuning overrides, event presentation, and package identity.

First proving target

Ship one complete themed app from the platform, then create a second tiny theme pack to prove the engine can generate a separate unlinked game without rewriting systems.

Practical implication

The repo should optimize for repeatable app generation, clean theme boundaries, and shared tests. The player never sees a theme switcher.
